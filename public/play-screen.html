    When you have found a defect, select the line(s) it is in. Then press the add button in the right corner below. It will be added to your defect list, from there you can select the reason for the defect.

    Remember to make use of your checklist and guide. If you are not sure about the type of defect that you have found or how to select it, you should refer back to the guide. -Bilsen Team

    This level will be your first real taste of the game. This level consists of a simple java class with code smells and even defects. Go and find them! As always select the error reasons carefully and reasonably in order to get more points.

    Things are getting more serious. In this level you are reviewing a very simple arithmetic operation implemented on arrays however in order to get a high score you need to pay close attention to the operations performed and their validity.

    Congratulations for making it this far! You are now dealing with a program performing matrix operations. While working with matrices things can always go wrong and defect can hide under your nose for a long time. So be careful.

    False Positives
    Instances where you think you have found a defect when there isn't any are false positives. False positives are unwanted in the code review process. You should double check your defect list before submitting to avoid false positives.

    Evolvability Defects
    Semantic duplication
    There are two sections in the code that are not the same but they perform the same task.
    Semantic dead code
    A piece of code that the program executes which does not perform any meaningful task.
    Duplication
    The exact replica of this code exists in the project.
    Consistency
    When something is achieveable with multiple ways or styles, you should select one of them and use only that for consistency. For example if you are naming your variables as camel case, you should not change it arbitrarily.
    Dead Code
    A piece of code in the program which is never executed.
    Bracket
    Unnecessary or inconsistent usage of brackets.
    Indentation
    Indentation of a certain code piece is not compliant with the rest of code.
    Blank line
    Empty lines usually indicate a separation of functionality thus they are commonly founded in places where single responsibility principle is violated.
    Long line
    Unnaturally long lines of code makes programs harder to read and debug, they might also make you miss code reuse opportunities.
    Naming
    Inconsistent or irrelevant naming.
    Comments
    Comments that are completely unnecessary or wrong.
    Immutable
    An immutable object is an object that can not be changed once its created.
    Void Parameter
    Null never should be a parameter to an object since it does not contribute to anything. The right way is to make 2 versions of the method one with the object parameter and the second with no parameters.
    Element Type
    Suboptimal usage of element (variable) type assignments.

    Functional Defects
    Function Call
    Improper or unnecessary function call, we should extend this to include middle man functions or multi responsibility methods.
    Parameter
    Too many or unnecessary parameters, wrong choice of parameters, perhaps even null parameters.
    Variable initialization
    Unnecessary dummy initializations, maybe some variable should have been declared as final or vica versa.
    Memory management
    Very common in some languages, not so common in others. Very hard to observe in languages where manual memory allocation is not allowed.
    Data and Resource Manipulation
    Defects that are due to mistakes made with data, variables, or other resources' initialization, manipulation, and release. Such as incorrect/suboptimal variable initializations and forgetting to release memory space after use.
    Compare
    Any errors made while utilizing "<",”>””=” type of operators are of this type.
    Compute
    These are the errors which are mathematics related rather than programming.
    Algorithm/Performance
    Obvious design choices that makes the program “significantly” inefficient memory or speed wise.
